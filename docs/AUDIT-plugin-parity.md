## Hand-maintained files (drift risk)

- `.claude-plugin/plugin.json` — name/description/author/keywords/hooks. The description hard-codes "18 specialized agents, 5 process discipline skills"; counts are not derived from `data/` and have drifted before.
- `.claude-plugin/README.md` — overlaps with top-level `README.md`; neither is generated from the other.
- Plugin skill set (5 of 36) is encoded by directory existence, not by an explicit allow-list. Adding a process skill requires creating both a `data/skills/` source and a `.claude-plugin/skills/<name>/` directory.

## OpenCode equivalent

`.opencode/` contains: `agents/*.md` (21 files, includes `lead.md` and a `test-agent.md` not in `data/` or `.claude-plugin/`), an `AGENTS.md` 16-byte stub, `package.json` declaring `@opencode-ai/plugin@1.4.9` (no `name`/`version`), and a committed `node_modules/`. It is **not** generated by `agent_notes.commands.build` and is not referenced by `build-plugin.sh` or `scripts/release`. `diff` confirms `.opencode/agents/coder.md` differs from `dist/opencode/agents/coder.md` — it's a local dev scaffold, not a distributable plugin.

OpenCode's plugin format (`@opencode-ai/plugin` SDK) is a JS/TS module with hooks — agent-notes does not produce one. End-user OpenCode install is `agent-notes install` writing to `~/.config/opencode/` (manual via the Python package).

## Config command in plugin

`plugin.json` exposes only `agents`, `skills`, and a `SessionStart` hook (cats `~/.claude/agent-notes-context.md`). No `commands` block, no slash command for `agent-notes config`/`doctor`/`update`/`list`. Plugin-only users must `pip install agent-notes` for the full CLI. Documented intentionally in `.claude-plugin/README.md` "Full install" section — known limitation, not a bug.

## Action items for Phase 8.B/8.C

1. Generate `plugin.json` `name`/`description`/`keywords`/`author` from `pyproject.toml` in `build-plugin.sh` (see `AUDIT-pypi-metadata.md`).
2. Replace the implicit "5 hand-curated process skills" rule with an explicit allow-list (e.g. `data/plugin/claude.yaml` listing vendored skills/agents) consumed by `build-plugin.sh`.
3. Decide `.opencode/`'s role. Either rename (`dev/.opencode/`) so it stops masquerading as a plugin, or extend `build-plugin.sh` with an OpenCode parity step that emits a real `@opencode-ai/plugin` package.
4. If the plugin should expose `agent-notes config` as a slash command, add a `commands/` block to `.claude-plugin/` sourced from `data/commands/`.
5. Generate `.claude-plugin/README.md` from a template under `data/templates/` to keep marketplace copy in lock-step.

## OpenCode plugin generation — deferred

OpenCode plugin generation is out of scope for Phase 8.B/C. The `.opencode/` directory is a local dev scaffold (see "OpenCode equivalent" above) — it is not a packaged plugin and does not follow the `@opencode-ai/plugin` SDK format. Before attempting to generate an OpenCode plugin, confirm the exact plugin format and hook API with the OpenCode upstream (`@opencode-ai/plugin`). Generating a JS/TS module with the wrong shape would be wasted work and may need to be regenerated after the format stabilises.
